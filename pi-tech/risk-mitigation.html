Software Development Lifecycle Controls

1. Secure Development Lifecycle (SDL)

Requirements: Formal security and quality requirements in every sprint/phase.

Design: Threat modelling for new features (STRIDE methodology).

Code: Peer review mandatory for production code; static analysis (SAST) on every commit.

Test: Automated unit tests (≥80% coverage target); integration testing; regression suite.

Deploy: Staged rollout — dev → staging → canary (1%) → gradual rollout.

Monitor: Real-time error tracking (Sentry/Datadog); rollback capability.

CrowdStrike lesson: Content configuration updates need same testing rigour as code updates.

2. Change Management & Deployment Controls

  • Change Advisory Board (CAB) for all production deployments
  • Deployment freeze windows (financial close periods, critical customer events)
  • Phased/canary deployments — validate with small user cohort before global rollout
  • Automated rollback triggers: error rate threshold → automatic rollback
  • Separation of duties: developer cannot deploy directly to production without approval
  • Audit trail: all deployments logged with approver, time, version

Key: CrowdStrike bypassed canary testing for content updates — market now requires change management controls to cover ALL update types.

3. Testing Standards

  • Unit testing: Minimum 70–80% code coverage (SonarQube or equivalent)
  • Integration testing: API contracts tested before deployment
  • Load testing: Production traffic simulation (AWS, JMeter)
  • Security testing: SAST (static code analysis) + DAST (dynamic analysis) + IAST in CI/CD pipeline
  • Penetration testing: Annual third-party pen test minimum; quarterly for financial services clients
  • User acceptance testing (UAT): Formal sign-off by client before go-live for custom development

4. Disaster Recovery & Business Continuity

  • Recovery Time Objective (RTO): Clearly defined per service tier (e.g., Tier 1 critical: RTO 1hr, RPO 15min)
  • Recovery Point Objective (RPO): Backup frequency aligned to RPO
  • Regular DR testing: Quarterly tabletop + annual full DR test
  • Geographic redundancy: Multi-region deployment for critical services
  • Incident playbooks: Pre-defined response procedures for common outage scenarios
  • Client communication protocol: SLA-defined notification windows (e.g., P1 = notify within 15 minutes)

Telecom F5 Specific Controls

1. Network Resilience

  • Redundant routing: No single point of failure in core network
  • N+1 capacity planning minimum; N+2 for critical paths
  • Automatic failover: BGP routing failover; sub-60-second convergence
  • Geographic diversity: Separate physical routes for redundant circuits
  • Power resilience: N+1 UPS; generator backup for all PoPs
  • Regular resilience testing: Simulate planned outages; validate failover paths

2. Emergency Services (999/112/911) Controls

  • Separate, isolated emergency call routing — not on shared commercial infrastructure
  • Priority queuing for 999/112/911 calls
  • Regular testing: Ofcom requirement to test emergency call connectivity quarterly
  • PSAP direct interconnect: No intermediaries in emergency call path where possible
  • Regulatory compliance: Ofcom General Condition C5 (UK) — mandatory emergency access
  • FCC 911 reliability rules (US)

Failure to route a single 999 call resulting in death = catastrophic liability + criminal investigation potential.

3. Billing System Controls

  • Billing audit: Automated cross-checks of call records vs charges
  • Systemic billing error detection: Anomaly detection on billing runs (e.g., charges >3σ from customer baseline)
  • Pre-bill review: Sample check before monthly bill run
  • Refund protocols: Rapid identification and compensation of overcharging
  • Ofcom Billing Code of Practice: Dispute resolution procedures mandatory

Billing errors affecting large customer bases = aggregate PI exposure (every customer a potential claimant).

4. SLA Management

  • SLA documentation: Defined, measurable SLAs in every customer contract
  • Monitoring: Real-time availability monitoring (Nagios, PRTG, Zabbis)
  • Automated SLA breach alerts: Escalation before credit threshold reached
  • Credit calculation: Automated, auditable SLA credit calculation
  • Customer communication: Proactive outage notifications before SLA breach
  • Contractual remedy vs PI: Ensure SLA credits fully satisfy client in most scenarios; PI exposure only where credits insufficient

AI/ML Risk Controls (Emerging)

1. AI Model Validation

  • Pre-deployment testing: Validation dataset separate from training data
  • Accuracy benchmarks: Defined minimum accuracy thresholds by use case
  • Bias testing: Protected characteristic fairness metrics
  • Hallucination testing: Adversarial prompting; out-of-distribution input testing
  • Model cards: Documentation of model capabilities, limitations and known failure modes
  • Third-party audit: Independent model validation for high-risk AI applications (EU AI Act)

2. AI Output Governance

  • Human-in-the-loop: Mandatory human review for high-stakes AI decisions (medical diagnosis, credit decisions, legal advice)
  • AI disclaimers: Clear disclosure when AI-generated content is provided to end users
  • Confidence scores: AI outputs include uncertainty/confidence indicators
  • Audit logging: All AI decisions logged for explainability and regulatory audit
  • Feedback loops: User correction mechanism to improve model; track error rates

3. Generative AI in Professional Services

  • AI use policy: Documented policy on employee use of GenAI (ChatGPT, Copilot) for client-facing work
  • Verification requirement: AI-generated legal, financial or technical content must be verified by human professional before delivery
  • Citation checking: AI-cited sources (cases, regulations, standards) independently verified
  • No hallucinated outputs policy: Clear prohibition on delivering AI output containing unverified facts

4. EU AI Act Compliance

  • Risk classification: Map all AI systems against EU AI Act risk tiers (unacceptable, high-risk, limited-risk, minimal-risk)
  • High-risk systems: Conformity assessment; technical documentation; human oversight mechanism; logging/audit
  • CE marking equivalent: Registration in EU AI Act database before market deployment
  • Penalties: Non-compliance fines up to 3% of global annual turnover
  • Applies to: AI in safety components of critical infrastructure, employment decisions, credit scoring, law enforcement, medical devices

Top 10 Risk Mitigation Actions

1
Implement phased/canary deployment for ALL production updates — including content and configuration updates (CrowdStrike lesson)
2
Negotiate limitation of liability clauses in every client contract; aim for 12-month fees cap as maximum exposure
3
Conduct annual penetration testing and quarterly automated security scanning on all client-facing systems
4
Define and document RTO/RPO for all service tiers; test disaster recovery quarterly; do not rely on untested recovery plans
5
For financial services clients: treat system failure scenarios as catastrophic; apply additional testing gates before deployment
6
For telecom (F5): isolate emergency call routing on separate, tested, redundant infrastructure; Ofcom/FCC compliance non-negotiable
7
Notify PI insurer immediately on awareness of any client claim, threat of claim, or circumstance likely to give rise to a claim
8
Document all AI model capabilities, limitations and validation results before deploying AI in client-facing services
9
Review client contracts annually for uncapped liability clauses; PI underwriters regard uncapped contracts as highest loading factor
10
Maintain incident response plan with client notification procedures; test with tabletop exercise at least annually
